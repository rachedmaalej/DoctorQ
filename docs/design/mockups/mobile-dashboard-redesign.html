DoctorQ Mobile Dashboard Redesign

Three alternative approaches based on user needs analysis

User Needs Analysis

visibility What I Need to See (Information Hierarchy)

  1. 1 Who's next? - The patient I'm about to call (most critical)
  2. 2 Who's currently with the doctor? - Current consultation status
  3. 3 How many are waiting? - Queue length at a glance
  4. 4 The rest of the queue - Full list for context

touch_app Primary Actions (Frequency of Use)

  1. 1 Call Next Patient - Used 20-40x per day (most frequent)
  2. 2 Add Patient - Walk-ins, phone appointments (~10x/day)
  3. 3 View Patient Status Link - Help confused patients (~5x/day)
  4. 4 Remove Patient - No-shows, cancellations (~3x/day)

Key Insight

The QR code is for patient onboarding, not daily operations. It should be accessible but not dominate the screen. The receptionist's primary loop is: See who's next → Call them → Repeat.

Option A

Queue-First

Prioritizes the queue list above all else. Stats are condensed into a single row. "Call Next" is integrated into the first queue item.

Recommendation

Option C (Split Focus) is recommended because it:

  • check_circle Keeps the most critical info (current patient + next patient) always visible without scrolling
  • check_circle Places the primary action (Call Next) prominently in the fixed area - one tap to advance the queue
  • check_circle Provides full queue access for context while not cluttering the main view
  • check_circle Moves secondary features (QR, settings) to a bottom bar - accessible but not distracting

Key improvement over current design: The QR code is demoted to a bottom tab (used once per patient onboarding, not core workflow). The "Call Next" button is elevated from a floating FAB to an always-visible primary action in context with the patient it will call.
